**60 Days of AI Challenge | Day 07**

## What I Worked On Today

Today I focused on building a practical decision framework for using Claude effectively across my daily workflow — coding, DSA practice, AI development, and career prep (resumes, GitHub READMEs, interview prep).

Instead of defaulting to one model for everything, I mapped out how to match **model tier** and **effort level** to the actual difficulty of the task.

## Key Takeaways

### Model Tiers
| Model | Best For |
|---|---|
| **Haiku** | Quick syntax checks, casual Q&A, fast lookups |
| **Sonnet** (daily driver) | DSA/competitive programming, debugging, README writing, research, resume drafts |
| **Opus** | Complex AI architecture decisions, hard multi-file bugs, high-stakes resume/interview prep |

### Effort Levels
- **Low** — quick syntax checks, simple fixes
- **Standard** (default) — everyday debugging, writing, explanations
- **High** — architecture decisions, tricky bugs, real interview prep
- **Max** — rare, high-stakes moments only (e.g. final portfolio polish)

### Core Lesson
> **Match the tool to the task — not the other way around.**

Using a heavier model/effort setting than a task needs wastes time and usage limits. Using a lighter one than a hard task needs risks lower-quality output. The goal is a workflow that stays fast for 80% of tasks and escalates depth only when the cost of a wrong answer is high.

## Personal Workflow I Adopted

```
Planning → DSA Practice → Coding → Debugging → AI Architecture
   → Documentation → Research → Resume/Interview Prep → Reflection
```

Sonnet handles most of this pipeline. Opus is reserved for genuinely hard debugging, architecture calls, and high-stakes career deliverables. Haiku covers quick, low-effort lookups.

## Why This Matters

As I build more AI-integrated projects (like CivicSphere AI) and continue daily DSA/CP practice, being deliberate about *how* I use AI tools — not just *that* I use them — is becoming as important as the technical skills themselves.
